# SRAホットスワップ機能におけるセキュリティリスクと攻撃手法

## 概要
SRA（Synaptic Routing Architecture）におけるシナプスの動的ホットスワップ機能は、極めて高い柔軟性と拡張性をもたらす一方で、稼働中のシステム外部から動的に実行モジュール（重みファイル）を読み込むという特性上、新たなセキュリティ脅威のベクトルを生み出します。本ドキュメントでは、想定される攻撃手法とその防御策について整理します。

## 想定される攻撃手法（脅威ベクトル）

### 1. 任意コード実行（Pickle/PyTorch Serialization Exploit）
**攻撃の概要:** 
PyTorchの標準的なモデル保存・読み込み機能（`torch.save` / `torch.load`）は、内部的にPythonの `pickle` モジュールを使用しています。攻撃者が悪意のあるPythonコードを埋め込んだ `.pt` ファイルを作成し、それを本番環境にホットスワップさせた場合、ファイルをロードした瞬間にGPUサーバー上で任意のシステムコマンドが実行される（RCE: Remote Code Execution）危険性があります。
**影響:** システムの完全な乗っ取り、データ流出、破壊工作。

### 2. 悪意のある重みの注入（Model Poisoning / Backdoor Injection）
**攻撃の概要:** 
攻撃者がシナプスの重みファイル自体を、特定のトリガー（特定の単語や入力パターン）に反応して誤った出力や機密情報を出力するように改ざんされた（バックドアが仕込まれた）シナプスと差し替えます。
**影響:** 例えば、「翻訳シナプス」に細工を施し、特定の企業名が含まれる文章が入力された時だけ意図的にネガティブな翻訳を出力させる、あるいはパスワード入力をバイパスさせる等のステルス性の高い攻撃が可能になります。

### 3. ルーター・キーの偽装によるルーティング・ハイジャック
**攻撃の概要:** 
プラグインとして提供されるサードパーティ製シナプスをホットスワップする環境において、悪意のあるシナプスが「機密情報処理シナプス」や「認証シナプス」と同じ「特徴ベクトル（Key）」を偽装してシステムに登録される攻撃です。
**影響:** ルーターは騙されて、本来セキュアなシナプスに送るべきユーザーの機密データ（個人情報など）を悪意のあるシナプスにルーティングしてしまい、データの窃取や改ざんが行われます。

### 4. スワップ・スラッシングによるサービス拒否攻撃（DoS / DDoS）
**攻撃の概要:** 
シナプスを外部ストレージから動的にGPUメモリへロードする際、PCIeバスの帯域やVRAMの割り当てに負荷がかかります。攻撃者が、システムに保存されている数千種類のシナプスを絶え間なく切り替えるよう要求する特殊なリクエストを大量に送信します。
**影響:** ホットスワップ処理のオーバーヘッドによりシステム全体がI/O待ち状態（スラッシング）に陥り、正規のユーザーへの応答ができなくなる（DoS攻撃）。

## 推奨される防御策（Mitigation Strategies）

1. **安全なフォーマットの採用（`safetensors`の義務化）**:
   - `pickle` 依存を脱却し、Hugging Faceが提唱する `safetensors` フォーマットのみをロード許可とすることで、任意コード実行（RCE）を完全に防ぐ。あるいは `torch.load(..., weights_only=True)` を徹底する。
2. **シナプスの暗号署名と検証**:
   - 信頼できる機関（または自社の鍵）でデジタル署名されたシナプスファイルのみをホットスワップ可能とし、ロード前に署名検証を行う（改ざん防止）。
3. **ホットスワップのレート制限（Rate Limiting）**:
   - 単位時間あたりのシナプス入れ替え回数に上限を設け、スラッシング攻撃によるDoSを防ぐ。
4. **ルーターキーの衝突検知と権限分離**:
   - 新規シナプスを登録する際、既存の重要なシナプスのKeyと類似度が高すぎる場合は登録を拒否する、あるいは実行環境（サンドボックス）を分けるアーキテクチャを導入する。
