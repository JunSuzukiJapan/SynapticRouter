# マルチタスク学習計画 2: 異種ドメインの言語モデリング (Code + Math + Text)

## 目的
Mixtralなどの巨大なMoEモデルで採用されているアプローチをSRAで模倣し、全く文法や語彙の異なるドメインの系列データ（プログラムコード、数式、自然言語）を同時に学習させます。ドメインごとにシナプスが分担され、干渉を防ぎながら学習できるかを検証します。

## データセット
小規模なトイ・データセットを用意します。
1. **Code**: 簡単なPythonのコードスニペット（例: `def add(a, b): return a + b`）
2. **Math**: 数式や計算式（例: `\int x^2 dx = \frac{1}{3} x^3`）
3. **Text**: 簡単な英語または日本語のテキスト（例: `The quick brown fox jumps over the lazy dog.`）

文字レベル（Character-level）の予測タスクにするか、簡単なBPEトークナイザ（例: `tiktoken`）を使用してトークン化します。

## 入出力フォーマット
通常のNext-token prediction（自己回帰的な言語モデリング）として学習させます。
明示的なタスク指定トークンは設けず、モデルが「入力されている系列のパターン」から自律的にドメインを推定し、適切なシナプスへルーティングすることを期待します。

- 入力: `<BOS> d e f   a d d ( a ,`
- ターゲット: `e f   a d d ( a ,   b`

## 実装ステップ

1. **データローダーの実装**:
   3つの異なるテキストファイル（`code.txt`, `math.txt`, `text.txt`）を読み込み、ランダムにサンプリングしてチャンク（固定長の系列）を生成するデータローダーを作成します。

2. **モデルのインターフェース変更**:
   現在のSRAはエンコーダ・デコーダ的な動作（`x` と `y_in` の分離）をしていますが、これを標準的なデコーダオンリー言語モデル（Decoder-only LLM）の形式に変更します。系列全体を入力とし、1つ先のトークンを予測する構造にします。

3. **学習の実行**:
   `MoESRAModel` を用いて学習を行います。

4. **クラスタリング分析**:
   テスト時に、コードを入力した際のシナプス使用頻度ベクトル、数式を入力した際の使用頻度ベクトルを収集し、PCAやt-SNEでプロットします。ドメインごとにルーティングのクラスタが形成されるかを確認します。

## 評価基準
- **パープレキシティ (Perplexity)**: 全ドメインにわたってクロスエントロピー損失が順調に低下するか。
- **ドメイン特化シナプスの発見**: 「コードの予約語（def, return）を予測する際によく発火するシナプス」のような、意味的な役割を持ったモジュールが自然発生しているか。
