# SRA GPU最適化アーキテクチャの比較・検証レポート

## 1. 背景と課題

オリジナルの Synaptic Routing Architecture (SRA) では、シナプスの評価を `nn.ModuleList` を用いたシーケンシャルな `for` ループで実行していました。これは PyTorch の GPU (CUDA / MPS) 実行環境において、以下の理由により深刻なボトルネックを引き起こしていました。

- **Host-Device同期の頻発**: 各シナプスの計算のたびにカーネル発行と同期が発生し、GPUの並列計算能力を活かせない。
- **計算グラフの肥大化**: VRAMの消費量が大きく、学習・推論共に遅延が大きい。

この課題を解決するため、3つの異なるGPU最適化アプローチを設計・実装し、ベースライン（Transformer / MLP）と比較・検証を行いました。

---

## 2. 検証した3つのアプローチ

### 案1: Batched SRA (`BatchedSRAModel`)
- **アーキテクチャ**: 全てのシナプス（例: 16個）の重みを1つのテンソル `(num_synapses, dim, hidden)` にまとめ、`torch.einsum` を用いて全シナプスを並列計算。
- **特徴**: オリジナルの「Self-Attentionを持つシナプス」の数学的挙動を完全に維持しつつ、ループを排除。
- **結果**: 速度が約4.5倍向上し、VRAM消費も大幅に削減（351MB → 55MB）。ただし「選ばれなかったシナプス」も計算するため、純粋なスパース計算ではない。

### 案2: MoE SRA (`MoESRAModel`)
- **アーキテクチャ**: シナプスから Self-Attention を取り除き、純粋なMLPエキスパートに変更。代わりにブロックの先頭に共有の `MultiheadAttention` を追加して系列コンテキストを保持。
- **特徴**: トークンごとに Top-k のエキスパートの重みを `self.w1[expert_idx]` などの高度なインデックス参照を用いて動的に抽出（Gather）し、`torch.bmm` で計算。
- **結果**: Batched SRAと同等の速度（約4.1倍向上）を維持しながら、**検証した全モデル中で最も低い Validation Loss** を記録。精度・速度のバランスが極めて優れている。

### 案3: Seq SRA (`SeqSRAModel`)
- **アーキテクチャ**: トークン単位ではなく「入力系列（シーケンス）全体」に対して Top-k のシナプスを選択し、選択したバッチに対してのみシナプスを評価。
- **特徴**: 評価回数を極限まで減らす設計だったが、GPU環境では「`nonzero()`等を用いた動的なバッチサイズの切り出しとマスキング」が原因で、シナプスごとに CPU-GPU の同期（ブロック）が発生。
- **結果**: 実行時間がオリジナルSRAよりもさらに約5倍遅くなる結果（1473秒）となった。ユニファイド・メモリのApple Silicon (MPS) 環境であっても、この「同期のオーバーヘッド」は解消されないことが判明した。

---

## 3. ベンチマーク結果

`copy` タスク（シーケンス長: 4〜10, バッチサイズ: 128, steps: 100）でのローカルGPU（Mac / MPS）によるパフォーマンステストの結果は以下の通りです。

| Architecture | Final Val Loss | Seq Accuracy | Total Time (s) | Max VRAM (MB) |
|--------------|----------------|--------------|----------------|---------------|
| **SRA (k=2)**    | 2.1782 | 0.001 | 304.99 | 351.5 |
| **Batched SRA**  | 2.5725 | 0.000 | **66.86** | 55.5 |
| **MoE SRA**      | **2.0645** | 0.001 | **73.47** | 104.5 |
| **Seq SRA**      | 2.1171 | 0.001 | 1473.13 | 43.1 |
| Transformer  | 2.1082 | 0.001 | 58.77 | 23.5 |
| MLP          | 2.3021 | 0.000 | 13.43 | 3.5 |

---

## 4. 結論と今後の方向性

検証の結果、**「MoE SRA」が速度と精度の観点で最も優れたアプローチである** ことが確認されました。
動的なテンソルサイズの変更（Seq SRAの敗因）を避け、固定サイズのマスク処理や重みの動的抽出（MoEの勝因）に徹することが、PyTorch環境下でのGPUパフォーマンスを最大化する鍵となります。

今後のSRAの発展においては、以下の理由から **MoE SRA** のアーキテクチャをベースラインとして研究・開発を進めることを推奨します。

1. **高い学習効率**: 共有Attention + MLPエキスパートの構成は、従来のLLMにおける標準的なMoE手法と相性が良く、精度低下を防げる。
2. **高速・省メモリ**: ループを廃し、真にルーティングされた部分のみのスパースな重みを抽出して計算するため、スケールアップが容易。
