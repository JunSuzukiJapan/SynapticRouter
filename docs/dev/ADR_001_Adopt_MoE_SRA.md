# Architecture Decision Record (ADR): MoE SRA の採用

## 決定事項
SRA (Synaptic Routing Architecture) の標準的な最適化・スケールアップ向けアーキテクチャとして、**MoE SRA (`MoESRAModel`) を採用することを決定**しました。

## 背景と課題
オリジナルの SRA 実装は、シナプス（小規模ネットワーク）単位での高度な専門化を目指し、各シナプス内部に「独立した Self-Attention」と「MLP」を内包する設計でした。しかし、この構造を PyTorch などの GPU 環境で実行するにあたり、以下の致命的なボトルネックが判明しました。

1. **Host-Device 同期のオーバーヘッド**: 
   各トークンが異なるシナプスにルーティングされるため、選ばれたトークンだけを動的に切り出して Self-Attention を計算しようとすると、GPUとCPU間で同期（ブロック）が頻発し、学習が非現実的なほど遅延する（Seq SRAの実験で実証済み）。
2. **無駄な計算とメモリ消費**: 
   同期を避けるために全シナプスを並列計算して最後にマスクをかける手法（Batched SRA）は高速化に成功したが、「選ばれなかったシナプスの Attention と MLP もすべて計算する」という無駄が発生し、真のスパース計算ではなく、スケールアップ時の VRAM 消費が懸念された。

## 採用理由（MoE SRAの優位性）

上記の問題を解決するため、SRAの構造を数学的に変更した「MoE SRA」を設計・テストし、以下の理由からこれを採用しました。

### 1. 「真のスパース計算」の実現
MoE SRA では、シナプス内部から Self-Attention を取り除き、**ブロックの先頭に全トークン共有の Attention を1つ配置**しました。これによりシナプスは「純粋なMLPエキスパート」となり、過去の文脈（シーケンス順序）を気にすることなく、**選ばれたシナプス（重み）だけを動的に抽出（Gather）して掛け算する** ことが可能になりました。無駄な計算を一切行わない「真のスパース計算」を実現しています。

### 2. 計算速度と VRAM 効率の圧倒的改善
同期オーバーヘッドを起こさず、無駄な計算もしないため、オリジナルSRAと比較して **約4倍の高速化** を達成しました。また、Batched SRA のように巨大な全並列テンソルを展開しないため、VRAM の消費量も極めて低く抑えられており、モデルの巨大化に耐えうる設計です。

### 3. 精度（Validation Loss）の向上
当初懸念された「シナプスごとに独自の Attention を持たなくなることによる精度低下」は発生しませんでした。むしろ、**共有 Attention でグローバルな文脈を把握し、スパースな MLP エキスパートで局所的なパターン処理に特化する** という役割分担（現在の最先端LLMにおける標準的な Mixture of Experts と同じ構造）が極めてうまく機能し、比較した全モデル中で **最も低い Validation Loss と高い安定性** を記録しました。

## 今後の影響
- SRAのコアな実験やアルゴリズムの検証は、引き続きオリジナルに近い「リファレンス実装（`sra_reference.py`）」を用いて直感的に行います。
- パフォーマンスが要求される大規模な学習や、実際のタスク適用においては、今回採用を決定した「MoE SRA実装（`sra_gpu_models.py`）」を標準として利用・拡張していきます。
