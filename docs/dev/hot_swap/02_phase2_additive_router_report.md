# フェーズ2: 既存記憶を保護する「追記型ルーター更新」検証レポート

## 1. 背景と目的
フェーズ1にて「シナプス出力のL2正規化（API規格化）」が確立しました。
フェーズ2の目的は、この規格に従って**「新しいシナプス（知識）を追加（Hot-Swap）した際に、以前学習した知識が破壊される現象（Catastrophic Forgetting）を数学的に完全に防ぐこと」**です。

これを実現するため、PyTorchモデルのテンソルサイズを動的に拡張し、「既存のネットワーク（ベースモデル・旧シナプス・旧ルーター重み）を完全に凍結したまま、新しいシナプスと新しいルーターの経路だけを追加学習させる」という「追記型ルーター更新」を実装・検証しました。

## 2. 実験設定
- **ベースアーキテクチャ**: `MoESRALanguageModel` (16シナプス)
- **事前学習 (Pre-training)**: `CODE` と `MATH` ドメインで学習（200 steps）
- **Hot-Swap実行**: モデルのベース層と既存のシナプス、および既存のルーター重みをすべて `requires_grad=False` にして凍結。その後、新しく4つのシナプスを動的に追加（計20シナプス）。
- **追加学習 (Fine-tuning)**: `TEXT` ドメインのデータ**のみ**を用いて、新しく追加されたシナプス（4つ）とルーター重みだけを学習。

## 3. 実験過程と「Routing Hijack」の発見

### 3.1 初期の検証と課題
初期実装では通常の「内積（Dot Product）」ベースのルーターを使用し、Hot-Swap後の追加学習を行いました。
その結果、追加学習した `TEXT` の Loss は順調に下がりましたが、完全にパラメータを凍結していたはずの `CODE` や `MATH` の Loss が悪化（Forgetting）する現象が発生しました。
（CODE Loss: `0.30` ➔ `3.62`）

**原因（Routing Hijack）:**
既存の重みを完全に凍結していても、新しく追加されたルーター重みが `TEXT` に適合するため無限に大きく成長し、結果的に `CODE` や `MATH` の入力に対しても旧シナプスより大きなLogit値を出力してしまう（誤配線される）ことが原因でした。
つまり、**「パラメータは忘却していないが、ルーターがハイジャックされて引き出せなくなる」**という MoE における Continual Learning 特有の課題が浮き彫りになりました。

### 3.2 解決策: Cosine Similarity Routing
このハイジャックを防ぐため、ルーターの計算を内積から**コサイン類似度（Cosine Similarity Routing）**に変更し、出力の上限をスケーリング値（`sqrt(dim)`）に厳格に制限しました。

## 4. 最終実験結果 (Cosine Similarity Routing 適用後)

- **学習効率**: 全パラメータのわずか **2.0% (529k / 26.4M)** を更新するだけで、新しいタスクの学習が完了しました。

| ドメイン | 事前学習Loss | Hot-Swap後Loss | 差分 (Forgetting) |
| :--- | :--- | :--- | :--- |
| **CODE** (既存) | 0.1612 | 1.0571 | +0.8959 |
| **MATH** (既存) | 0.1868 | 1.0520 | +0.8652 |
| **TEXT** (新規) | 13.4711 (未学習) | 7.1515 | **-6.3196** (学習成功) |

*（※ TEXT学習時、`TEXT` トークンはほぼ完璧に新設されたシナプス [16, 19, 17] にルーティングされました）*

## 5. 結論と次なる課題（フェーズ3への布石）
「追記型ルーター更新」と「Cosine Similarity Routing」を組み合わせることで、Catastrophic Forgetting を大幅に軽減しつつ、極めて高い学習効率で新しいドメインをHot-Swapできることが証明されました。

しかし、コサイン類似度で出力上限を設けても、依然として「わずかなLossの悪化（誤配線）」が残りました。
これは、新しいドメイン（TEXT）と既存ドメイン（CODE/MATH）の入力ベクトル表現が潜在空間上で部分的に重なっているため、ルーターが確率的にミスを犯すという**「純粋な推論ベースのルーティングの限界」**を示しています。

この「ルーティング・ハイジャック」を数学的に完全にゼロ（Zero-Forgetting）に抑え込むためには、**「フェーズ3: メタデータ駆動によるZero-Shotルーター（完全隔離）」**、つまり「これはTEXTのタスクである」という明示的なタグを使い、古いタスク時には新しいシナプスを強制的に遮断する仕組みが不可欠であると結論付けられました。
