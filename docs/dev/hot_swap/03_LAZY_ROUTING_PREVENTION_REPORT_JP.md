# LAZY ROUTING PREVENTION: ルーターの怠けを防ぐアプローチの検証

## 1. 概要
Synaptic Routing Architecture (SRA) などの Mixture of Experts (MoE) モデルでは、学習中に一部のエキスパート（シナプス）ばかりが選ばれ、大半のシナプスが使われなくなる「Lazy Routing（ルーターの怠け）」や「Representation Collapse」と呼ばれる問題が頻発します。
本レポートでは、128シナプスという巨大なキャパシティを持つSRAモデルに対して、Lazy Routingを防ぐための2つのアプローチを導入し、その効果を検証した結果をまとめます。

## 2. 検証したアプローチ
1. **Baseline**: 現状のSRA実装（`load_balance_loss` のみ適用済み）。
2. **Capacity Limit (ハード制約)**: 1つのシナプスがバッチ内で処理できるトークン数に上限を設け、溢れたトークンは強制的にDrop（切り捨て）する。
3. **Entropy Control (ソフト制約)**: ロス関数を操作し、「バッチ全体としては全シナプスを均等に使う（エントロピー最大化）」かつ「個々のトークンは迷わず1つのシナプスを強く選ぶ（エントロピー最小化）」ように誘導する。
4. **Both**: 上記両方の組み合わせ。

## 3. 実験結果と分析

各条件で1000エポックのマルチタスク学習を行った結果、以下のメトリクスが得られました。

| 条件 | Routing Entropy (Max 4.85) | Dead Synapse Ratio (%) |
| :--- | :--- | :--- |
| 1. Baseline | 4.688 | 11.7% |
| 2. +Capacity Limit | 4.776 | 17.2% |
| 3. +Entropy Control| 3.901 | 43.0% |
| 4. +Both | 4.132 | 30.5% |

*(※ Routing Entropyは高いほど均等。Dead Synapse Ratioは使用率が極端に低いシナプスの割合で、低いほど健全)*

### 3.1. ベースラインの優秀性の確認
最も驚くべき結果は、**対策を追加していない Baseline の段階で、すでに Routing Entropy が 4.69 と高く、死んだシナプスが 11.7% に抑えられていた**ことです。これは、初期実装から導入されていた `load_balance_loss`（使用率と均等分布とのMSEをとるロス）が極めて優秀に機能し、128個ものシナプスを効率よく分散利用できていることを証明しています。

### 3.2. Capacity Limit の副作用
Capacity Limitを追加すると、エントロピーは最大値近くまで上昇し、分布全体はより均等になりました。しかし、Dead Synapse は 17.2% へと微増しました。
これは、容量をオーバーしたトークンを単に「Drop（切り捨て）」する実装にしたため、人気シナプスへの一極集中は防げたものの、**「溢れた分を別の空きシナプスに再割り当てする」仕組みがなかった**ため、元々死んでいたシナプスが蘇生（再利用）されるには至らなかったことが原因です。

### 3.3. Entropy Control による「超特化」の罠
Entropy Controlを入れた結果、エントロピーは低下し、Dead Synapseは 43.0% へと大幅に悪化しました。
これは「個々のデータに対して迷わず1つのシナプスを選べ」というペナルティが強すぎた結果、ルーターが各タスクに対してごく少数の特定シナプスに完全に固定（超特化）してしまい、残りの半数近いシナプスが使われなくなってしまったためです。ソフト制約におけるハイパーパラメータ調整の難しさを示しています。

## 4. 結論と今後の課題

本実験を通じて、**「現在の SRA のベースライン実装は、一般的なMoEが苦労する Lazy Routing 問題をすでに高水準で回避できている」** という強みが実証されました。現状のアーキテクチャは非常に健全に動作しています。

今後のさらなる改善（Dead Synapse 0% の実現）に向けては、単純なロス操作やトークンの切り捨てではなく、**「容量を超過したトークンを、確率が2番目に高い空きシナプスへ転送する（Routing Fallback）」** という、より高度な動的ルーティング・アルゴリズムの実装が必要であるという明確な指針が得られました。
