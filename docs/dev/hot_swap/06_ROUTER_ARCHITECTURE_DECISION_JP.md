# SRAルーターアーキテクチャの標準仕様と設計方針

**日付:** 2026-05-19
**対象:** SRAネイティブ統合モデルにおけるルーターアーキテクチャの標準仕様の策定

## 1. 概要
Notebook 24で実施された「ルーターアーキテクチャ・ベンチマーク」の結果に基づき、SRAマザーボード構想（OSS LLMネイティブ統合）において使用するルーターモデルの標準仕様を定義します。

従来の「Mean Pooling + 単層Linear」構成で判明した、汎用トークンによるルーティング確率の希釈問題（確信度が 0.6〜0.7 に留まる現象）を解決するため、複数のアーキテクチャを比較検証しました。

## 2. ベンチマーク結果と考察
ベンチマーク（Notebook 24）では、以下の4つのアーキテクチャを比較しました。

| アーキテクチャ | 特徴 | パラメータ数 | 平均確信度 |
| :--- | :--- | :--- | :--- |
| **Baseline (Mean Pooling)** | 全トークンの平均ベクトルを入力 | 4,098 | 0.9997 |
| **A: Last-Token Router** | 最後のトークンベクトルのみを入力 | 4,098 | 1.0000 |
| **B: MLP Router** | Mean Pooling + 非線形隠れ層 | 1,050,114 | 0.9999 |
| **C: Attention Router** | トークンごとの重要度を重み付け | 6,147 | 0.9999 |

**考察:**
- 全てのアーキテクチャにおいて目標とする確信度（>0.99）を達成しました。
- 中でも **Last-Token Router** は、Baselineと全く同じ極小のパラメータ数でありながら、最速で Loss=0.0000・確信度 1.0000 に到達しました。
- 自己回帰型LLMは「最後のトークン」に文章全体の文脈を圧縮する性質を持つため、これをルーターの入力として直接利用することが、最も効率的かつ希釈を防ぐ手段であることが証明されました。

## 3. 標準アーキテクチャの決定 (Default Router)
上記の検証結果より、パラメータオーバーヘッドを極限まで削りつつ確実なルーティング（確信度 > 0.99）を実現するため、**「Last-Token Router」をSRAのデフォルト（標準）ルーターとして採用します。**

今後のSRAコア実装や新しいNotebookでは、明示的な指定がない限り、この Last-Token ベースのルーターを使用します。

## 4. 拡張ルーターのサポート設計 (Pluggable Router Architecture)
デフォルトは Last-Token Router としますが、SRAは「モジュラーAI」であることを重視し、**タスクの複雑さやユーザーの要件に応じて他のルーター構造も選択肢として利用・拡張できる（Pluggableな）設計**を維持します。

例えば、以下のようなケースでは別アーキテクチャのルーターをホットスワップ的に適用できるようにします。

- **MLP Router の活用:**
  複雑な非線形境界を学習する必要がある（非常に似通った複数の専門ドメインを見分ける等）タスクにおいて、パラメータ数の増加（約1MB）を許容できる場合。
- **Attention Router の活用:**
  プロンプトの途中に散りばめられた特定のキーワード（トリガーワード）がルーティングに決定的な役割を果たす場合（系列全体の意味を拾いつつ、特定単語を強調したい場合）。
- **ユーザー定義ルーター:**
  今後、Vision-Language Model (VLM) などのマルチモーダルモデルと統合する際に、画像パッチとテキストを統合して処理できる専用のルーターを後から注入できるように設計します。
