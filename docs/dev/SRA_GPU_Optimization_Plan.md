# SRA GPU Optimization Plan (次回の課題)

## 目的
現在のSynaptic Routing Architecture (SRA) は、Pythonの `for` ループを用いてシナプスを順次評価しているため、GPU（CUDAやMetal/MPS）の並列処理の恩恵を受けられず、実行速度に深刻なボトルネックを抱えている。
次回は、このGPUボトルネックを解消するための**3つの異なるアプローチを実装し、それぞれを比較検証**する。

---

## 比較検証する3つのアプローチ

### 案1: Batched（バッチ化）評価による並列計算
* **概要:** `nn.ModuleList` による個別のシナプス評価をやめ、全シナプスの重みパラメータを1つのテンソルにスタックする。`torch.vmap` やカスタムのバッチ行列積（BMM）を用いて、全16個のシナプスをGPUの1回の命令で同時並列に計算する。
* **メリット:** 現在の「Self-Attentionを持つシナプス」の数学的な挙動を一切変更せずに、純粋なスループットを数十倍に引き上げられる。
* **デメリット:** `Top-k` に選ばれていないシナプスの計算も行う（後でマスクして0にする）ため、真のスパース計算ではなく、VRAMの消費量が約16倍（全シナプス分）になる。
* **検証ポイント:** VRAM消費量の増加が実用範囲内に収まるか、および実装の簡便さ。

### 案2: MoE（Mixture of Experts）型のスパース計算へのアーキテクチャ変更
* **概要:** 現在の `TinySynapse` から Self-Attention の役割を取り除き、純粋な MLP（または極小のFeedForward層）に変更する。トークンごとに「選択されたシナプス」へデータをGather（集約）して計算し、計算後にScatter（分散）して元の系列に戻す。
* **メリット:** ルーティングされたシナプスのみを計算する「真のスパース計算」となり、VRAM効率と計算効率が極めて高い（現在のLLMにおける標準的なMoE手法）。
* **デメリット:** シナプス自身が系列全体の文脈（Attention）を持たなくなるため、Routerや前後の共有層で文脈処理を補うなどのアーキテクチャ変更が必要になる。
* **検証ポイント:** アーキテクチャ変更に伴う学習効率（精度）の低下がないか、または補えるか。

### 案3: Sequence-level（系列レベル）のルーティングへの変更
* **概要:** 現在の「トークン（文字）ごとにシナプスを選択する」方式を改め、「入力系列（文章やバッチ内の1サンプル）ごとに Top-k のシナプスを選択する」方式に変更する。
* **メリット:** 1つの文章に対して選ばれたシナプス（例: 2個）だけを動かせばよいため、ループ処理が激減し、Attentionの計算もそのまま維持できる。
* **デメリット:** トークン単位での細やかな専門化（例: 「動詞」はシナプスA、「名詞」はシナプスB）ができなくなり、系列全体のコンテキスト単位での専門化へとパラダイムが変化する。
* **検証ポイント:** 系列単位でのルーティングでも、SRAの強みである高いSample Efficiencyや構造的知能が維持されるか。

---

## 次回のアクションプラン
1. 上記3つのアプローチをそれぞれ `src/` 配下に独立したモデルクラス（例: `BatchedSRAModel`, `MoESRAModel`, `SeqSRAModel`）として実装する。
2. 作成済みの `src/compare_architectures.py` を拡張し、これらのモデルをGPU環境で実行する。
3. 学習の収束速度（Step vs Validation Loss）、実行速度（Total Time）、および VRAMピーク使用量を測定し、どのアプローチが SRA の今後の発展に最も適しているかを決定する。
