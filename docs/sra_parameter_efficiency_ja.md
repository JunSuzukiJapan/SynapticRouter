# SRA (Synaptic Routing Architecture) のパラメータ効率に関する考察

本ドキュメントでは、通常のDense Transformer（一枚岩のモデル）と比較した際に、SRAが「同じ品質を達成するために必要なパラメータ数を削減できるか」という問いについて、理論的な考察を行う。将来的な実験検証の仮説として位置づける。

---

## 1. 中心仮説

**マルチタスク・マルチドメイン学習において、SRAは同じ品質を達成するために必要な総パラメータ数がDense Transformerよりも少なくなる。**

その根本的な理由は、**パラメータ空間における「干渉（interference）」の排除**にある。

---

## 2. 理論的根拠

### 2.1 Dense Transformerにおけるパラメータの「妥協」

通常のDense Transformerでは、すべてのパラメータがすべてのタスク・ドメインに対して活性化される。マルチタスク学習（例: Code + Math + Text の同時学習）を行うと、各パラメータが複数ドメインの知識の**折衷表現（compromise）**に収束してしまう。

これにより以下の問題が生じる：
- 各ドメインにとって「最適ではないが、全体としてはマシ」な表現しか獲得できない
- 精度を上げるには、ドメイン間の干渉を吸収できるだけの**大きなパラメータ容量**が必要になる
- パラメータの一部が、本来不要な「他ドメインとの折り合い」のために消費される（＝無駄遣い）

### 2.2 SRAにおけるパラメータの「専門化」

SRAでは、ルーターが入力に応じて**タスクごとに異なるシナプスを割り当てる**ため、各シナプスは自分が担当するタスクにのみ特化できる。

- 各シナプスのパラメータは、1つのドメインだけを学習すればよいので、**小さな容量で高精度を達成**できる
- 共通の知識（例: Decision Transformerの実験で確認された「空間知覚」の共有モジュール）は**シナプスの自然な共有**によって冗長性なく表現される
- パラメータ空間に「妥協」が生じないため、**パラメータあたりの情報密度（利用効率）が高い**

### 2.3 概念的な比較

3つのドメイン（Code, Math, Text）を同時に学習させる場合を概念的に比較する：

| アーキテクチャ | パラメータの使われ方 |
|---|---|
| **Dense Transformer** | 全パラメータが3ドメインの折衷表現を保持。干渉を吸収するために巨大な容量が必要 |
| **SRA** | 共有ベース層（Embedding/Positional等）+ ドメインごとの極小シナプス。各シナプスは自分のドメインだけに特化するため小さくてよい |

例えるなら、SRAでは「100パラメータのシナプス × 3ドメイン ≈ 300パラメータ」でそれぞれに特化できるのに対し、Dense Transformerでは「1000パラメータの巨大な層」でようやく3ドメインの干渉を吸収できる、というイメージである。

---

## 3. 条件による場合分け

### 3.1 マルチタスク・マルチドメイン学習：【SRAが有利】

SRAのモジュール分離の恩恵が最大限に発揮される。ドメイン数が増えるほど、Dense Transformerとの効率差は拡大すると予想される。

- タスク間干渉がゼロ（Lesion実験で実証済み）
- 各シナプスが担当範囲に特化するため、少ないパラメータで高い専門性を達成
- ドメイン間で共有すべき知識は、ルーターが自律的にシナプスを共有することで冗長性を排除

### 3.2 シングルタスク学習：【ほぼ同等〜わずかにDenseが有利】

単一のタスク（例: シェイクスピアのテキスト生成のみ）を学習させる場合、SRAの「モジュール分離」の恩恵は限定的。

- ルーターがほぼ同じシナプスを選び続けるため、実質的にDense Transformerと同じ挙動になる
- ルーターのパラメータ（`synapse_emb`等）がオーバーヘッドとなる
- 同じ総パラメータ数なら、シングルタスクではDense Transformerがわずかに有利

### 3.3 推論時のアクティブパラメータ：【SRAが圧倒的に有利】

パラメータ効率とは別に、推論時に実際に計算に使用される「アクティブパラメータ数」ではSRAが圧倒的に有利：

```
Dense Transformer (7B):  推論時に7B全パラメータが毎回計算に参加
SRA (7B, k=2/16):       推論時に実際に計算するのは ~1B分のシナプスだけ
```

つまり「総パラメータ数は同等でも、推論時の計算量は数分の1」という状態が実現できる。

---

## 4. MoE文献との整合性

この仮説はMoE (Mixture of Experts) の研究結果とも整合する：

- **Mixtral 8x7B**: 総パラメータ46.7B、アクティブパラメータ12.9Bで、Llama 2 70B（Dense, 70Bパラメータ）と同等以上の性能を達成
- **Switch Transformer**: 同じ計算コスト（FLOPs）ではDense Transformerに比べて大幅な品質向上を報告

SRAはMoEをさらに細粒度にしたアーキテクチャ（シナプス単位での分離）であるため、この効果がより顕著に出る可能性がある。

---

## 5. 総合比較表

| 観点 | SRA vs Dense Transformer |
|---|---|
| **マルチタスクで同じ品質に必要な総パラメータ数** | ✅ SRAの方が少なくて済む可能性が高い |
| **シングルタスクで同じ品質に必要な総パラメータ数** | ⚠️ ほぼ同等か、わずかにDenseが有利 |
| **推論時のアクティブパラメータ（計算量）** | ✅ SRAが圧倒的に少ない |
| **学習時のVRAM消費** | ❌ SRAの方が多い（全シナプス保持が必要） |
| **パラメータあたりの学習効率（情報密度）** | ✅ SRAが高い（干渉がないので各パラメータが有効に使われる） |

## 6. 将来の実験検証案

本考察を実験的に検証するためのプランを以下に示す。

### 6.1 パラメータ数 vs 精度のスケーリング曲線

**目的**: 同じマルチタスクベンチマークにおいて、SRAとDense Transformerのパラメータ数に対する精度のスケーリング曲線を描き、「同じ精度を達成するために必要なパラメータ数」を定量的に比較する。

**手法**:
1. 3ドメイン（Code, Math, Text）のマルチタスク学習を、以下のパラメータ数で実施：
   - SRA: 1M, 5M, 10M, 50M, 100M パラメータ（シナプス数・シナプスサイズを変動）
   - Dense Transformer: 同じパラメータ数帯
2. 各規模でのValidation Loss / Accuracy を記録
3. 両者のスケーリング曲線を重ねてプロット

**期待される結果**: SRAのスケーリング曲線がDense Transformerよりも「左上」に位置する（＝少ないパラメータで同じ精度に到達する）。

### 6.2 ドメイン数に対する効率のスケーリング

**目的**: ドメイン数を2, 4, 8, 16と増やした場合に、SRAの効率優位がどう変化するかを検証する。

**仮説**: ドメイン数が増えるほど、Dense Transformerではパラメータの「干渉コスト」が指数的に増大するが、SRAではシナプス追加のみで線形にスケールできるため、効率差が拡大する。

### 6.3 情報密度の測定

**目的**: 学習後の各パラメータが実際にどれだけ「有用な情報」を保持しているかを測定する。

**手法**:
- Pruning（枝刈り）実験: パラメータを重要度の低い順に削除していき、精度劣化のカーブを比較
- SRAの方がパラメータの利用効率が高ければ、枝刈りに対する耐性が低い（＝すべてのパラメータが有効に使われている）はず

---

## 7. 結論

SRAのパラメータ効率の本質は、**「ルーターによるモジュール分離がパラメータ空間の干渉を排除し、各パラメータが自分の専門分野だけに集中できる」**点にある。これにより、特にマルチタスク・マルチドメイン学習において、Dense Transformerよりも少ない総パラメータ数で同等以上の品質を達成できるという仮説は、理論的に十分な根拠があると考える。

ただし、これはあくまで理論的考察であり、実験的な検証（特にセクション6のスケーリング曲線の比較）によって定量的に裏付ける必要がある。
